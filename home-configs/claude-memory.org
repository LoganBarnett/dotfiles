#+title:     Global Claude Instructions
#+language:  en

* Document format

Prefer org-mode (~.org~) for new documents unless markdown is strictly required
or the project was established by non-org-mode users (e.g. existing repo with a
~README.md~).

GitHub does not require markdown files for anything - org-mode can be used in
place of any markdown file.

* Prose conventions

When authoring prose — org-mode docs, markdown, multi-sentence code
comments, commit message bodies, plain text — put two spaces after
sentence-ending punctuation (~.~, ~!~, ~?~).  Single space inside a
sentence as normal.

This gives an unambiguous machine-readable sentence boundary, so
vim/evil sentence text objects (~dis~, ~cis~, ~)~, ~(~) work
reliably.  Single-space + period collides with abbreviations like
"Mr." or "e.g."; two-space terminators do not.  Vim's ~'cpoptions'~
~J~ flag formalizes the same convention.

When editing existing prose that uses single-spacing, match the
surrounding style rather than reflowing.  HTML/web output is exempt
— browsers collapse whitespace in rendered output regardless.

* Nix

** Explicit ~default.nix~ paths

Always use explicit ~default.nix~ suffixes in import paths (e.g.,
~callPackage ../derivations/foo/default.nix {}~).  Do not rely on Nix's
implicit directory-to-~default.nix~ resolution — it adds cognitive burden
and prevents editors from jumping directly to the file.

This applies to code in this repository.  External sources such as nixpkgs,
home-manager, and nix-darwin use the implicit form idiomatically; follow
their conventions when contributing to or copying patterns from those.

* Dev environment

Unless stipulated otherwise, non-trivial projects should be built in Rust.  Use
~/new-rust-project~ to scaffold them.

** Global tools

Never use tools that download or install software globally.  This is a Nix
environment; such tools (~bundler~, ~rustup~, ~brew~, and similar) would
either fail or cause destructive side-effects outside the Nix-managed
system.

Project-specific tools (~cargo~, ~go~, language runtimes, etc.) must live
in the project's ~flake.nix~ under ~devShells~.  The direnv integration
activates the devshell automatically; just run the tool directly.

** Command-line JSON parsing

Use ~jq~ for command-line JSON parsing.  It is available across all hosts.
Do not use ~python3~ — it is not reliably on ~PATH~ in this Nix environment.
