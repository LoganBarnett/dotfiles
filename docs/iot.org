#+title:     IoT Devices
#+language:  en

* Overview

This document covers IoT devices on the home network: how they're identified,
how they're provisioned onto the LAN, and how they're controlled locally.

The general posture for any IoT device on this network is:

- Provision onto the LAN once, ideally without the vendor's cloud app.
- After provisioning, block the device's egress to the WAN at the firewall.
  Local control keeps working; cloud heartbeats fail harmlessly.
- Pin a static DHCP lease so addresses don't drift.
- Prefer open-source local control (Home Assistant, openHAB, CLI tools)
  over vendor mobile apps.

* Stealth split heat pump

** Identification

The unit is sold as "Stealth" by Stealth Comfort Solutions (a 1HVAC brand),
but the hardware and firmware are Gree-OEM rebranded.  Confirmed by:

- Provisioning AP SSID format (=GR_AC_xxxxxxxx=).
- Error code map matches Gree exactly (e.g. H5 = IPM Protection trip).
- Vendor control app is Gree+, not a Stealth-specific app.
- LAN protocol is the Gree UDP/7000 protocol, including AES-ECB and
  AES-GCM variants for newer firmware.

For service work, Gree service manuals (e.g. RIO, Multi) are usually the
correct reference when the Stealth manual is thin.

** Provisioning runbook

Goal: get the unit off its temporary provisioning AP and onto the home
WiFi network so it can be controlled locally via ~gree-remote~.

*** 1. Put the unit in AP/pairing mode

On the OEM remote, hold ~MODE~ + ~TURBO~ for ~10 seconds until the unit
beeps twice.  The IDU will start broadcasting its provisioning AP.

The AP will appear nearby as an SSID like ~GR_AC_xxxxxxxx~ (the trailing
characters are derived from the unit's MAC).

*** 2. Connect the host to the unit's AP

From the host that has ~gree-remote~ installed (currently scandium), join
the unit's AP.  The well-known default password is:

#+begin_example
12345678
#+end_example

This is the same on every Gree-OEM unit out of the box — it's not a
per-device secret.  The host will get a DHCP lease in ~192.168.1.x~, with
the unit itself reachable at ~192.168.1.1~.

*** 3. Push the home WiFi credentials

#+begin_src sh :exports code
gree-remote wlan --ssid '<home-ssid>' --psw '<home-password>'
#+end_src

The ~wlan~ subcommand sends an unencrypted UDP packet to ~192.168.1.1:7000~
with the WiFi credentials.  A successful response looks like:

#+begin_example
WLAN configuration accepted: {'t': 'ret', 'r': 200, 'mac': '<device-mac>'}
#+end_example

The unit reboots onto the home network and the provisioning AP disappears.
Note the ~mac~ field in the response — it's the device MAC, useful if you
need to drive ~bind~ directly later (e.g. across a VLAN where broadcast
discovery doesn't reach).

The ~wlan~ subcommand is provided by our fork of ~tomikaa87/gree-remote~
(see [[*Implementation notes][Implementation notes]]); it is not in upstream.

*** 4. Verify on the LAN

Reconnect the host to the home WiFi, then discover the unit and capture
its bind details:

#+begin_src sh :exports code
gree-remote search -b 192.168.x.255
#+end_src

(Replace ~192.168.x.255~ with the actual home subnet broadcast.)

Search broadcasts a scan, decrypts each response, and immediately invokes
~bind~ for every device found.  Our fork tries both ECB and GCM encryption
on bind — V3+ firmware reports a GCM-capable version string but in practice
only accepts ECB-keyed bind requests, so the fallback matters.

Capture four things from the output:

- the unit's IP
- its MAC ID
- the per-device key (16 chars, e.g. ~Fbmtho7no8Q22aL3~)
- *the encryption mode that succeeded* (printed in the "Binding device:
  ... encryption: ECB|GCM" line)

The encryption mode matters for every subsequent ~get~ / ~set~ — passing
the wrong ~-e~ flag silently times out the same way the wrong-mode bind
did.  V3+ firmware in this network has been observed to bind in ECB; older
units may bind in GCM.

Pin a static DHCP lease for the unit's MAC on the router so the IP doesn't
drift.

** Re-provisioning onto a different network

If the unit has already been provisioned and you want to move it to a
different SSID, *always reset the WiFi state on the unit before pushing
new credentials*.  Skipping this step can leave the unit in a "stuck"
state where it responds to ~search~ but refuses to bind or accept commands
because of cached binding state — see upstream issue
[[https://github.com/tomikaa87/gree-remote/issues/54][tomikaa87/gree-remote#54]].

To reset:

1. Power on the unit.
2. Hold ~MODE~ + ~TURBO~ on the remote for 10 seconds until the unit
   beeps twice.  This drops it back to AP/unbound mode and the
   ~GR_AC_xxxxxxxx~ AP reappears.
3. Then run the provisioning runbook above against the new SSID.

For a unit that has *never* been provisioned (e.g. a fresh install), this
reset step is unnecessary — pressing MODE+TURBO simply puts it into AP
mode for the first time.

** Local control

Once the unit is on the LAN and bound, ~gree-remote get~ and ~gree-remote
set~ read and write parameters.  Pass the same ~-e~ encryption mode that
the bind succeeded with:

#+begin_src sh :exports code
gree-remote get -c <ip> -i <mac> -k <key> -e ECB Pow Mod SetTem TemSen
gree-remote set -c <ip> -i <mac> -k <key> -e ECB SetTem=21
#+end_src

Common parameter names include ~Pow~, ~Mod~, ~SetTem~, ~WdSpd~, ~TemSen~
(see ~gree.py~ source for the full list).

This is control plane only — outdoor-unit sensors (compressor current, IPM
temperature, discharge temperature) are not exposed over the LAN protocol;
for those, the IDU's hidden self-check mode on the remote is the path.

** Implementation notes

- ~gree-remote~ is packaged in this repo at
  [[file:../derivations/gree-remote/default.nix]] and made available via
  [[file:../agnostic-configs/iot-utils.nix]].
- Our fork at ~LoganBarnett/gree-remote~ (branch ~wlan-subcommand~) carries
  three patches not in upstream:
  1. ~wlan~ subcommand for AP-mode WiFi provisioning.
  2. ~bind~ subcommand for invoking the bind handshake without a broadcast
     scan (useful across VLANs).
  3. Compatibility patches for V3+ firmware: search now falls back to
     ~pack['mac']~ when ~cid~ is empty, and ~bind~ retries the opposite
     encryption mode on timeout.  Both are needed because V3+ units
     report GCM in scan responses but only respond to ECB-keyed binds.
- See [[file:../../nix-config-private/tasks.org]] for the upstream-PR
  status of these patches.
