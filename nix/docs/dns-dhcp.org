#+title:     DNS and DHCP
#+language:  en

* DHCP

** Host Identification

Identifying device types even when a host does not advertise its hostname.
Techniques are grouped by how much network traffic they generate.

*** Passive

These require no additional probes; data is already present or can be captured
without initiating contact.

- *MAC OUI lookup*: The first three octets of a MAC address identify the
  hardware manufacturer (Apple, Samsung, Raspberry Pi Foundation, etc.).  The
  IEEE publishes the OUI registry; offline databases are available via
  ~nmap~'s ~nmap-mac-prefixes~ or the ~ieee-data~ package.  MAC addresses are
  already available as Prometheus labels from the DHCP lease textfile exporter.

- *DHCP vendor class identifier (option 60)*: Clients optionally include a
  string identifying their OS or device type, e.g. ~android-dhcp-14~,
  ~MSFT 5.0~, ~dhcpcd-9.x~.  dnsmasq logs this when ~log-dhcp~ is enabled.
  Coverage is incomplete but precise when present.

- *DHCP parameter request list (option 55)*: The ordered list of DHCP options
  a client requests is OS-specific and forms the basis of DHCP fingerprinting.
  Matching against a database such as [[https://fingerbank.org][Fingerbank]] yields OS and device type
  with good confidence.  dnsmasq logs this alongside option 60 when
  ~log-dhcp~ is enabled.

*** Semi-Passive

These listen for unsolicited broadcasts or multicasts without initiating
connections to individual hosts.

- *mDNS / DNS-SD* (~avahi-browse -a~): Devices such as phones, smart TVs,
  Chromecast, Sonos, and Apple devices announce service types (e.g.
  ~_airplay._tcp~, ~_googlecast._tcp~) even without a DHCP hostname.

- *SSDP / UPnP* (multicast ~239.255.255.250:1900~): IoT devices, smart TVs,
  and Windows hosts advertise device descriptions via the Simple Service
  Discovery Protocol.

- *NetBIOS name service* (UDP 137): Windows and Samba hosts broadcast
  NetBIOS names on the local segment.

*** Active

These initiate contact with individual hosts and generate traffic that the
target can observe.

- *nmap OS fingerprinting* (~nmap -O~): Analyzes TCP/IP stack behavior (TTL,
  window size, TCP options) to classify the remote OS.  Accurate, but
  produces a port scan that the target host sees.

- *p0f*: Passive TCP/IP stack fingerprinting.  Sits on the wire and classifies
  hosts from traffic they generate, without sending anything itself.  Labelled
  active here because it typically requires a span port or promiscuous capture,
  which is a deliberate infrastructure change.
